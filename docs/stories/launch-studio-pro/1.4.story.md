## Story

**As a** desenvolvedor do Launch Studio PRO,
**I want** adicionar os campos de plano premium (`premium_plan_slug`, `premium_since`, `premium_expires_at`) à tabela `users` e criar a tabela `user_premium_limits` com os limites por plano (Starter/Pro/Agency),
**so that** o sistema possa aplicar limites de projetos e regenerações por plano sem apagar dados existentes (PRD §1, §10; params §11.10).

## Acceptance Criteria

1. Campos `premium_plan_slug text`, `premium_since timestamptz`, `premium_expires_at timestamptz` adicionados à tabela `users` via migration Prisma (PRD §8.1, §1).
2. Índice composto `(user_id, is_premium)` criado em `users` (PRD §8.4).
3. Tabela `user_premium_limits` criada com campos: `id`, `plan_slug` (starter/pro/agency), `max_projects_per_month` (3/25/-1), `max_regenerations_per_month` (10/-1/-1), `max_concurrent_generations` (1/5/-1), `created_at` — onde -1 representa ilimitado.
4. Dados semente (seed) inseridos em `user_premium_limits` com os 3 planos conforme PRD §10: Starter (3 proj/mês, 10 regen), Pro (25 proj/mês, ilimitado), Agency (ilimitado, ilimitado).
5. Campo `premium_plan_slug` aceita apenas valores `starter | pro | agency | null`.
6. Modelo Prisma `UserPremiumLimits` gerado com tipagem correta.
7. `npx prisma validate` e `npm run typecheck` passam sem erros.

## Dev Notes

### Contexto brownfield
- Tabela `users` já existe em `prisma/schema.prisma`. Esta story ADICIONA colunas — não recria.
- Verificar que não existe já campo `premium_plan_slug` antes de adicionar (brownfield check).
- **CORREÇÃO (Decisão R3-C, 2026-06-18):** O Stripe webhook em `app/api/stripe/webhook/route.ts` NÃO popula `is_premium` — ele processa apenas pagamentos de anúncios (`checkout.session.completed` com `metadata.adId`). O campo `is_premium` é atualmente ÓRFÃO: nenhuma rota o seta. A ativação premium via Stripe (checkout de assinatura → webhook → `is_premium=true` + `premium_plan_slug` + `premium_since`) é coberta pela story 1.7, que é pré-requisito para que o gate premium funcione de ponta a ponta para assinantes. Esta story (1.4) apenas cria a estrutura de dados; a story 1.7 cria o fluxo de ativação.

### Limites por plano (PRD §10)

| Plano | Projetos/mês | Regenerações/mês | Concorrentes |
|-------|-------------|-----------------|-------------|
| Starter | 3 | 10 | 1 |
| Pro | 25 | ilimitado (-1) | 5 |
| Agency | ilimitado (-1) | ilimitado (-1) | ilimitado (-1) |

### Convenção: -1 = ilimitado
Usar `-1` como sentinel para ilimitado, não `NULL`, para facilitar comparação no código.

### Dependência
Depende de 1.1 (tabelas base criadas). Não depende de 1.2 ou 1.3.

**Nota de dependência downstream:** A story **1.7** (Integração Stripe → Ativação Premium) depende desta story para existência dos campos `premium_plan_slug`, `premium_since` e `premium_expires_at` em `users`. Sem 1.7, `is_premium` permanece órfão e assinantes não ativam o gate premium.
