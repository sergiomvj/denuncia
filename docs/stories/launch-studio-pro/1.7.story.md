## Story

**As a** desenvolvedor do Launch Studio PRO,
**I want** criar o fluxo de checkout de assinatura premium via Stripe e o webhook que ativa `users.is_premium = TRUE` junto com `premium_plan_slug` e `premium_since`,
**so that** usuários que assinam um plano premium (Starter/Pro/Agency) tenham `is_premium` setado corretamente e o gate `resolveSextouToolsProUser` de `lib/sextou-tools/auth.ts` passe a liberá-los (PRD §7 Stripe, §8.1, §10; Decisão R3-C de 2026-06-18).

## Acceptance Criteria

1. Rota `POST /api/stripe/checkout/subscription` cria uma Stripe Checkout Session do tipo `subscription` com `line_items` correspondentes ao plano escolhido (`starter | pro | agency`) e `metadata.userId` + `metadata.planSlug` (PRD §7 Stripe, §10).
1a. O fluxo usa **Stripe Checkout HOSPEDADO** (hosted checkout): o cliente é redirecionado para `session.url` (página da própria Stripe) onde insere os dados de cartão. NENHUM dado de cartão trafega ou é armazenado no servidor da plataforma (escopo PCI delegado à Stripe). NÃO usar Stripe Elements/cartão embedded no domínio próprio nesta story.
1b. A session é criada com `success_url` (retorno pós-pagamento, ex.: `/sextou-tools-pro/launch-studio?premium=success&session_id={CHECKOUT_SESSION_ID}`) e `cancel_url` (retorno em caso de desistência, ex.: a página `acesso`/upgrade) — PRD §7.
1c. É criado um componente client REUTILIZÁVEL `PagamentoStripe` em `components/sextou-tools-pro/pagamento-stripe.tsx` que pode ser chamado por QUALQUER app Premium da suite. O componente recebe o `planSlug` (e props opcionais como `returnTo`/label), chama a rota de checkout de assinatura e redireciona o navegador (`window.location.href = data.url`) para a Stripe Checkout hospedada. Segue o padrão do componente existente `components/payment-options.tsx` (loading/erro/`window.location.href`).
1d. A tela de upgrade (story 1.5) e quaisquer outros apps Premium consomem o componente `PagamentoStripe` — nenhum app reimplementa a chamada de checkout localmente (ponto único de pagamento, espelhando o ponto único de gate `resolveSextouToolsProUser`).
2. O webhook em `app/api/stripe/webhook/route.ts` trata o evento `checkout.session.completed` quando `metadata.planSlug` estiver presente (distinguindo de pagamento de anúncio que usa `metadata.adId`).
3. Ao receber o evento de assinatura confirmada, o webhook seta em `users` via Prisma: `is_premium = true`, `premium_plan_slug = metadata.planSlug`, `premium_since = now()` (PRD §8.1 campos adicionados na story 1.4).
4. Campo `premium_expires_at` é populado com `null` para planos de assinatura recorrente (sem data de expiração fixa), ou com a data da próxima cobrança se disponível no evento Stripe.
5. Usuário com `is_premium = TRUE` (setado pelo webhook) é liberado pelo gate `resolveSextouToolsProUser` de `lib/sextou-tools/auth.ts` — teste end-to-end do fluxo completo.
6. Evento Stripe `customer.subscription.deleted` ou `invoice.payment_failed` reverte `is_premium = false` e `premium_plan_slug = null` (cancelamento/expiração de assinatura).
7. Webhook é idempotente: processar o mesmo evento duas vezes não duplica dados nem lança exceção.
8. A rota de checkout reusa o padrão existente em `lib/stripe.ts` (`getStripe`, `createCheckoutSession` ou helper equivalente) — não reimplementar lógica Stripe.
9. `npm run lint` e `npm run typecheck` passam sem erros.
10. Teste manual: simular evento `checkout.session.completed` (com Stripe CLI ou `stripe trigger`) → verificar que `users.is_premium = true` e `users.premium_plan_slug` correto são setados no banco.

### Gate end-to-end após esta story
Com `is_premium = true` setado pelo webhook, o helper `resolveSextouToolsProUser` em `lib/sextou-tools/auth.ts` (linha ~111: `!result.user.hasActiveAds && !result.user.isPremium`) retornará `kind: "ok"` e o gate liberará o acesso. Nenhuma alteração no helper é necessária — a lógica OR já está implementada.

### Mecanismo de pagamento — Stripe Checkout HOSPEDADO (decisão do owner, 2026-06-18)
O fluxo de "tornar-se Premium" usa o carrinho/checkout **hospedado pela Stripe**: a rota cria uma Checkout Session em modo `subscription` e retorna `session.url`; o frontend redireciona o navegador para essa URL, e o cliente preenche os dados de cartão **na plataforma da Stripe**, não no domínio próprio.
- Vantagem: o servidor da plataforma nunca recebe/armazena PAN do cartão → escopo PCI fica com a Stripe (SAQ-A), sem necessidade de Stripe Elements/embedded.
- `success_url` traz o cliente de volta ao app (a ativação do `is_premium` ocorre via webhook `checkout.session.completed`, fonte de verdade — não confiar apenas no redirect de sucesso).
- `cancel_url` retorna à tela de upgrade/`acesso` sem ativar nada.
- O fluxo espelha o padrão de checkout de anúncios já existente em `app/api/stripe/checkout/route.ts` (mesma mecânica de hosted checkout), apenas com `mode: 'subscription'` e `metadata.planSlug`.

### Componente compartilhado `PagamentoStripe` (decisão do owner, 2026-06-18)
O pagamento Premium deve ter um ÚNICO ponto de entrada reusável por todos os apps Premium da suite — espelhando a centralização do gate em `resolveSextouToolsProUser`. Em vez de cada app criar seu próprio botão de checkout, todos importam `components/sextou-tools-pro/pagamento-stripe.tsx`.
- Padrão de referência: `components/payment-options.tsx` (já faz `fetch` da rota Stripe e `window.location.href = data.url` com estados de loading/erro). Reusar essa mecânica, trocando o endpoint para `/api/stripe/checkout/subscription` e o payload para `{ planSlug }`.
- O componente é "burro" quanto ao gate: ele só inicia o checkout. A liberação de acesso continua sendo decidida pelo helper de gate central após o webhook ativar `is_premium`.
- Local: `components/sextou-tools-pro/` (mesma pasta dos demais componentes Premium PRO).

### Stripe Price IDs
Os Price IDs dos planos premium (`STRIPE_PRICE_STARTER` etc.) devem ser criados no Stripe Dashboard antes da implementação. Para desenvolvimento, usar Price IDs de ambiente de teste (test mode). Documentar no `.env.example` com valores placeholder.

Esta story é o pré-requisito para que o caminho `is_premium` do gate funcione de ponta a ponta. Sem ela, assinantes não têm `is_premium = true` e o acesso via assinatura premium está quebrado. A via `hasActiveAds` já funciona (aprovação de anúncio seta `hasActiveAds=true` em `app/api/admin/ads/[id]/approve/route.ts`).
