## Story

**As a** desenvolvedor do Launch Studio PRO,
**I want** criar a tabela `premium_access_logs` e o middleware `premiumGate` aplicado a todas as rotas do app,
**so that** apenas usuários com acesso PRO (assinatura premium ativa OU anúncio ativo) acessem o Launch Studio PRO, com tentativas registradas para auditoria (PRD §0, §11.1; Decisão R3-C de 2026-06-18).

## Acceptance Criteria

1. Tabela `premium_access_logs` criada com campos: `id`, `user_id`, `app_slug` (default `'launch-studio-pro'`), `attempted_at`, `granted` (boolean), `ip_address`, `user_agent` (PRD §0 SQL do gate).
2. Índice `(user_id, attempted_at)` criado em `premium_access_logs` (PRD §8.4).
3. Middleware `premiumGate` implementado server-side (Next.js middleware ou API route guard) que verifica acesso PRO via helper central `resolveSextouToolsProUser` de `lib/sextou-tools/auth.ts`.
4. O gate libera acesso quando `is_premium = TRUE` OR `hasActiveAds = TRUE` — ambas as condições são verificadas pelo helper central; o middleware NUNCA checa um flag isoladamente (Decisão R3-C).
5. Usuário sem acesso PRO é redirecionado para `/acesso?app=launch-studio-pro` ou equivalente — tela que comunica as DUAS vias de acesso (assinatura premium OU anúncio ativo).
6. Toda tentativa de acesso (granted=true e granted=false) é registrada em `premium_access_logs` com `ip_address` e `user_agent`.
7. O middleware REUSA `resolveSextouToolsProUser` de `lib/sextou-tools/auth.ts` para obter o usuário e resolver o gate — nenhuma lógica de gate duplicada no middleware (STORIES-BREAKDOWN §1 Reuso).
8. Todas as rotas sob `/app/launch-studio-pro/**` passam pelo `premiumGate`.
9. A página de acesso existente (`app/sextou-tools-pro/acesso/page.tsx`) tem seu texto atualizado para listar explicitamente as duas vias de liberação: (a) assinatura premium ativa, (b) anúncio ativo na plataforma.
10. `npm run lint` e `npm run typecheck` passam sem erros.
11. Teste manual: usuário sem acesso PRO ao acessar qualquer rota do app é redirecionado e tentativa é logada com `granted=false`.
12. Teste manual: usuário com `is_premium = TRUE` acessa normalmente; tentativa logada com `granted=true`.
13. Teste manual: usuário com `hasActiveAds = TRUE` e `is_premium = FALSE` acessa normalmente; tentativa logada com `granted=true`.

## Dev Notes

### Contexto brownfield
- `lib/sextou-tools/auth.ts` contém o helper central `resolveSextouToolsProUser` — REUSAR (não reimplementar gate).
- A lógica OR (`hasActiveAds || isPremium`) já está implementada na linha ~111 do helper: `if (!result.user.hasActiveAds && !result.user.isPremium)` → retorna `kind: "forbidden"`. O middleware deve confiar nesse helper e nunca duplicar a verificação.
- A página de acesso existente em `app/sextou-tools-pro/acesso/page.tsx` menciona apenas "anúncios ativos" como via de acesso. Deve ser atualizada para também mencionar assinatura premium como segunda via.
- Stripe/webhook existentes em `app/api/stripe/*` processam pagamento de anúncios — não modificar nesta story.

### Assinatura do helper central (lib/sextou-tools/auth.ts, ~linha 104)

```ts
export async function resolveSextouToolsProUser(): Promise<SextouToolsProUserLookupResult>
// Retorna:
//   { kind: "ok", user }          → acesso liberado (is_premium OR hasActiveAds)
//   { kind: "forbidden", user }   → autenticado mas sem acesso PRO
//   { kind: "unauthorized" }      → não autenticado
//   { kind: "db-unavailable" }    → falha de DB
```

### Padrão de implementação conceitual

```ts
// middleware/premiumGate.ts (ou em cada route handler do launch-studio-pro)
import { resolveSextouToolsProUser } from "@/lib/sextou-tools/auth"

export async function premiumGate(req: NextRequest) {
  const result = await resolveSextouToolsProUser() // REUSE — gate com OR centralizado
  if (result.kind === "unauthorized") {
    return redirect("/login?next=/launch-studio-pro")
  }
  if (result.kind === "forbidden") {
    await db.premiumAccessLog.create({ ...granted: false... })
    return redirect("/sextou-tools-pro/acesso?app=launch-studio-pro")
  }
  if (result.kind === "ok") {
    await db.premiumAccessLog.create({ ...granted: true... })
    // prosseguir
  }
}
```

### Decisão R3-C (PO, 2026-06-18)
Acesso PRO = `is_premium OR hasActiveAds`, sempre via `resolveSextouToolsProUser`. O PRD §0 ("exclusivo is_premium = TRUE") é reinterpretado como "premium dá acesso", não "apenas premium dá acesso". Anunciantes ativos também recebem acesso de cortesia — comportamento já implementado no helper central.

### CRITICAL — Gate server-side
PRD §16: "Não permitir acesso a usuários não-premium sob nenhuma circunstância." Nunca verificar flags de acesso apenas no client — toda verificação deve ser server-side via `resolveSextouToolsProUser`.

### Dependência
Esta story depende de 1.1 (tabela `premium_access_logs` usa FK para `users`).
