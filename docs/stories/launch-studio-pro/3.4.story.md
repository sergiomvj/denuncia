## Story

**As a** usuário premium do Launch Studio PRO,
**I want** que minhas respostas da entrevista sejam salvas automaticamente por etapa e que eu possa retomar de onde parei,
**so that** eu não perca progresso se fechar o browser ou precisar continuar depois (PRD §8.2).

## Acceptance Criteria

1. Cada resposta de etapa é salva (upsert) em `launch_interview_answers` com `project_id`, `step_number`, `step_slug` e `answers` (jsonb) ao avançar para a próxima etapa ou ao sair do wizard.
2. Ao reabrir o projeto/wizard, o sistema carrega todas as respostas salvas de `launch_interview_answers` por `project_id` e pré-preenche os campos correspondentes.
3. O wizard reinicia na última etapa respondida (step_number mais alto com resposta salva).
4. O `status` do projeto em `launch_projects` é atualizado para `'interviewing'` ao salvar a primeira resposta da entrevista.
5. API route `GET /api/launch-studio-pro/projects/[id]/interview` retorna todas as respostas salvas do projeto como `{ stepNumber: answers }`.
6. API route `PATCH /api/launch-studio-pro/projects/[id]/interview/[step]` faz upsert da resposta da etapa.
7. Respostas são validadas no server: `answers` deve ser um objeto JSON não-vazio.
8. `npm run lint` e `npm run typecheck` passam.

## Dev Notes

### Upsert Prisma

```ts
await prisma.launchInterviewAnswers.upsert({
  where: {
    projectId_stepNumber: { projectId, stepNumber: Number(step) }
  },
  update: { answers, stepSlug, coachTipShown },
  create: { projectId, userId: currentUser.id, stepNumber: Number(step), stepSlug, answers, coachTipShown }
});
```

Nota: constraint única `(project_id, step_number)` é necessária para o upsert. Verificar se 1.1 já a criou; se não, criar migration adicional.

### Status update

```ts
// Ao salvar primeira etapa, atualizar status do projeto
const answersCount = await prisma.launchInterviewAnswers.count({ where: { projectId } });
if (answersCount === 0) { // primeira resposta
  await prisma.launchProject.update({
    where: { id: projectId },
    data: { status: 'interviewing' }
  });
}
```

### Dependência
Depende de 3.1 (wizard engine), 3.2 (componentes de etapa). Esta story formaliza a camada de persistência que 3.1 iniciou.

### Nota sobre 3.1 vs 3.4
Story 3.1 criou a API de save. Esta story formaliza o contrato completo (GET + PATCH + status update + constraint). Coordenar com o dev que implementou 3.1 para não duplicar código.
