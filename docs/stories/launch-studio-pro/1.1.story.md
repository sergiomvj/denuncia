## Story

**As a** desenvolvedor do Launch Studio PRO,
**I want** criar as tabelas `launch_projects`, `launch_interview_answers`, `launch_dossiers`, `launch_phases` e `launch_triggers_map` via migration do banco de dados,
**so that** o domínio PLF tenha persistência estruturada com os tipos, índices e soft delete definidos no PRD §8.2.

## Acceptance Criteria

1. Migration Prisma cria as tabelas `launch_projects`, `launch_interview_answers`, `launch_dossiers`, `launch_phases` e `launch_triggers_map` com todos os campos especificados no PRD §8.2.
2. `launch_projects` possui campo `deleted_at timestamptz` para soft delete (PRD §11.2).
3. Índices criados conforme PRD §8.4: `(user_id, status)` em `launch_projects`; `(project_id, step_number)` em `launch_interview_answers`; `(dossier_id, version)` em `launch_dossiers`; `(user_id, attempted_at)` em `premium_access_logs` (tabela criada em 1.2 mas índice documentado aqui).
4. Tipos de dados respeitam o PRD: `jsonb` para `answers`, `payload`, `copy_blocks`; `text[]` para `triggers_activated`; `numeric` para `fidelity_score`; `integer` para `version`, `step_number`, `intensity`.
5. `launch_projects.status` aceita valores: `draft | interviewing | planning | ready | launched | archived`.
6. `launch_phases.phase_slug` aceita valores: `pre_pre | plc1 | plc2 | plc3 | cart_open | cart_close`.
7. `launch_triggers_map.trigger_name` aceita valores conforme PRD §4.2 etapa 10: `authority | reciprocity | trust | anticipation | empathy | events | community | scarcity | social_proof`.
8. `npm run db:migrate` (ou equivalente Prisma) aplica a migration sem erros em ambiente local.
9. `npx prisma validate` passa sem erros após a migration.
10. `npm run typecheck` passa sem erros após geração do Prisma Client.

## Dev Notes

### Contexto brownfield
O projeto já utiliza Prisma com PostgreSQL/Supabase. O schema existente está em `prisma/schema.prisma`. Esta story adiciona modelos novos no mesmo schema — não modifica modelos existentes.

### Tabelas a criar (PRD §8.2)

```sql
-- launch_projects
CREATE TABLE launch_projects (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id uuid NOT NULL REFERENCES users(id),
  title text NOT NULL,
  slug text NOT NULL,
  target_audience text NOT NULL,
  initial_briefing text NOT NULL,
  status text DEFAULT 'draft',
  launch_type text,
  fidelity_score numeric,
  created_at timestamptz DEFAULT now(),
  updated_at timestamptz DEFAULT now(),
  deleted_at timestamptz
);
```

Replicar padrão para as outras 4 tabelas (ver PRD §8.2 completo).

### Modelos Prisma equivalentes
Usar `@map` para snake_case no banco e camelCase no TypeScript. Padrão observado nos modelos existentes (`SextouToolsProGeneration` em `lib/sextou-tools-pro/`).

### REUSE
- FK para `users(id)` já existe no schema — apenas referenciar.
- Padrão `created_at / updated_at` já adotado em toda a suite.

### Estratégia: domínio `launch_*` vs `premium_*`
[AUTO-DECISION] Tabelas `launch_*` para domínio PLF; tabelas `premium_*` para telemetria transversal da suite (criadas em E4). Razão: separação domínio/infraestrutura conforme STORIES-BREAKDOWN §8.

### Testing

- Testar migration em DB local com `npx prisma migrate dev --name launch_schema_foundation`
- Verificar que rollback manual funciona (drop das 5 tabelas sem erro)
- `npx prisma validate` deve retornar "Schema is valid"
- `npm run typecheck` sem erros de tipos gerados
