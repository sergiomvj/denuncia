## Story

**As a** desenvolvedor do Launch Studio PRO,
**I want** integrar Sentry (com fallback Logtail) para capturar erros e traces do app em produção,
**so that** falhas de geração, erros de API e problemas de runtime sejam monitorados com logs sanitizados e sem PII (PRD §7).

## Acceptance Criteria

1. Sentry SDK integrado e capturando erros não tratados no app Launch Studio PRO (PRD §7).
2. Fallback: se Sentry não configurado, usar Logtail ou apenas `console.error` estruturado.
3. Erros dos agentes LLM (falha de geração, timeout, schema inválido) capturados com contexto do `project_id` e `phase_slug`, mas sem `user_id` visível no trace (sanitização de PII).
4. API keys de provedores NUNCA aparecem nos traces do Sentry (integração com sanitizeLog de 4.5).
5. Trace de performance para runs de geração: `startTime` e `endTime` capturados por `premium_app_runs`.
6. `SENTRY_DSN` configurado via variável de ambiente do servidor.
7. `npm run lint` e `npm run typecheck` passam.

## Dev Notes

### Sentry beforeSend hook

```ts
// sentry.server.config.ts
Sentry.init({
  dsn: process.env.SENTRY_DSN,
  beforeSend(event) {
    // Sanitizar API keys nos extra/contexts
    if (event.extra) {
      event.extra = sanitizeObject(event.extra);
    }
    return event;
  }
});
```

### Dependência
Depende de 4.5 (sanitizeLog). Deve ser implementada antes ou junto ao deployment de produção.

### Testing

- Lançar erro intencional: capturado no Sentry (ou console.error sem Sentry)
- Verificar que trace não contém API keys
- `npm run typecheck` sem erros de tipos do Sentry SDK
