## Story

**As a** desenvolvedor do Launch Studio PRO,
**I want** bloquear a geração quando o usuário atinge os limites do seu plano (projetos, regenerações, concorrência) com modal de upgrade, sem apagar projetos existentes,
**so that** os limites de negócio sejam respeitados e o usuário seja incentivado a fazer upgrade (PRD §10, §12).

## Acceptance Criteria

1. Limite de projetos por plano verificado no CREATE de projeto (já implementado em 2.4) — esta story centraliza e consolida a lógica de verificação de limites.
2. Limite de regenerações por plano verificado antes de cada regeneração (de 6.6).
3. Limite de gerações simultâneas (Starter: 1, Pro: 5, Agency: ilimitado) verificado antes de iniciar nova run — se usuário já tem `N` runs `status='running'`, bloquear.
4. Modal de upgrade exibido em todos os casos de limite atingido, com mensagem específica por tipo de limite.
5. Projetos existentes NUNCA apagados ao atingir limite (PRD §10).
6. Função utilitária `checkUserLimits(userId, limitType): Promise<LimitResult>` criada em `lib/launch-studio-pro/limits.ts` para centralizar verificações.
7. `npm run lint` e `npm run typecheck` passam.

## Dev Notes

### Centralização de limites

```ts
// lib/launch-studio-pro/limits.ts
export type LimitType = 'projects' | 'regenerations' | 'concurrent';

export async function checkUserLimits(userId: string, type: LimitType): Promise<{
  allowed: boolean;
  limit: number;
  current: number;
}> {
  const user = await getUserWithPlan(userId);
  const limits = await getPlanLimits(user.premiumPlanSlug ?? 'starter');
  
  switch (type) {
    case 'concurrent': {
      const running = await countRunningRuns(userId);
      const max = limits.maxConcurrentGenerations;
      return { allowed: max === -1 || running < max, limit: max, current: running };
    }
    // ... outros casos
  }
}
```
