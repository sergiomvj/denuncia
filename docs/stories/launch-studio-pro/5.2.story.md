## Story

**As a** desenvolvedor do Launch Studio PRO,
**I want** definir o schema JSON completo do dossiê de lançamento e implementar validação da saída dos agentes contra esse schema,
**so that** toda saída de LLM seja estruturada e validável, com rejeição e retry automático em caso de saída inválida (PRD §9.2, Output JSON).

## Acceptance Criteria

1. Schema TypeScript do dossiê criado em `lib/launch-studio-pro/schemas/dossier-schema.ts` com todos os campos do PRD §9.2 (project, avatar, offer, sequence, triggers_map, calendar, joint_launch, warnings, fidelity_checklist).
2. Validação de schema implementada via Zod ou equivalente — retorna `{ valid: boolean, errors: string[] }`.
3. Ao receber output do LLM, o sistema tenta parsear como JSON e validar contra o schema.
4. Se output inválido (não-JSON ou não-conforme): registra o erro e tenta novamente (até 2 retries com prompt de correção).
5. Após 2 retries sem sucesso, a run é marcada como `failed` em `premium_app_runs` com detalhes do erro.
6. Schema exportado como tipo TypeScript para uso em todo o codebase.
7. Testes unitários cobrem: JSON válido, JSON inválido (campo faltando), não-JSON.
8. `npm run lint` e `npm run typecheck` passam.

## Dev Notes

### Schema JSON do dossiê (PRD §9.2 — transcrição fiel)

```ts
// Estrutura principal
export const DossierSchema = z.object({
  project: z.object({ title: z.string(), slug: z.string(), launch_type: z.string() }),
  avatar: z.object({
    profile: z.record(z.unknown()),
    pains: z.array(z.string()),
    dreams: z.array(z.string()),
    fears: z.array(z.string()),
    objections: z.array(z.string()),
    language: z.array(z.string()),
  }),
  offer: z.object({
    transformation: z.string(),
    core_product: z.record(z.unknown()),
    bonuses: z.array(z.unknown()),
    guarantee: z.string(),
    anchor_price: z.number(),
    launch_price: z.number(),
    scarcity_type: z.enum(['price_increase', 'bonus_removal', 'offer_ends', 'combined']),
  }),
  // ... sequence, triggers_map, calendar, joint_launch, warnings, fidelity_checklist
});
```

### Prompt de correção para retry

```ts
const correctionPrompt = `
Sua resposta anterior não é JSON válido ou está faltando campos obrigatórios.
Erros encontrados: ${errors.join(', ')}.
Retorne APENAS o JSON corrigido, sem texto adicional, seguindo o schema fornecido.
`;
```

### Dependência
Depende de 5.1 (system prompt base e contratos). O schema é o "contrato de saída" que complementa os contratos de entrada de 5.1.

### Testing

- Dossier válido completo: `validateDossier` retorna `{ valid: true }`
- Dossier sem `offer.scarcity_type`: erro descritivo
- `"não é JSON"` passado: erro de parse capturado sem crash
