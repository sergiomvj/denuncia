## Story

**As a** desenvolvedor do Launch Studio PRO,
**I want** implementar fallback automático por estratégia `same_modality_lower_cost` quando um provedor falha,
**so that** a geração continue mesmo com falhas de API, com toda tentativa registrada em `premium_llm_calls` (PRD §5, params §7.5).

## Acceptance Criteria

1. Para cada `phase_role`, existe uma cadeia de fallback definida (primário → fallback 1 → fallback 2) conforme PRD §5.
2. Ao receber erro de API (timeout, rate limit, 5xx), o roteador tenta o próximo provedor da cadeia automaticamente.
3. Máximo de 3 tentativas por chamada (primário + 2 fallbacks).
4. Cada tentativa (sucesso ou falha) é registrada em `premium_llm_calls` com `status` correto (success/failed/fallback) e `attempt_number`.
5. Se todos os fallbacks falharem, o roteador lança erro descritivo com detalhes das tentativas.
6. A estratégia `same_modality_lower_cost` é respeitada: fallback usa mesmo tipo de tarefa mas modelo/provedor mais econômico.
7. Testes unitários cobrem cenários: sucesso na 1ª tentativa, sucesso na 2ª, falha em todas.
8. `npm run lint` e `npm run typecheck` passam.

### Implementação de retry

```ts
async function callWithFallback(params: CallParams): Promise<CallResult> {
  const configs = await getFallbackChain(params.phaseRole);
  let lastError: Error;
  for (let i = 0; i < configs.length; i++) {
    try {
      const result = await callProvider(configs[i], params);
      await logLlmCall({ ...result, attemptNumber: i + 1, status: 'success' });
      return result;
    } catch (err) {
      await logLlmCall({ error: err, attemptNumber: i + 1, status: i === 0 ? 'failed' : 'fallback' });
      lastError = err;
    }
  }
  throw new AllProvidersFailedError(lastError);
}
```
